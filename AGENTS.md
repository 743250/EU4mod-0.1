# AGENTS.md

## 文件定位

`AGENTS.md` 只记录长期、全局、跨文件的协作规则和设计边界。

不要把这里当草稿本、排查日志、TODO 清单或提交说明。具体机制方案、未来剧情线、阶段性判断和下一步计划应放到 `docs/` 或代码附近注释中。

这里应该保留：

- 项目方向的极简边界，以及不能跨越的设计底线。
- 跨文件机制边界。
- EU4 脚本、编码、本地化的硬约束。
- 容易反复踩坑的经验。
- 修改后的最低验证标准。

这里不应该记录：

- 某次具体修改过程。
- 临时猜测、未验证方案和个人草稿。
- `docs/幕府系统设计文档.md` 中已经记录的核心功能目标、未来路线和具体机制方案。
- 细碎 key 清单、流水账式文件索引。
- 只影响单个文件的实现细节。

## 文档分工

根目录的 `AGENTS.md` 是 Codex 的 mod 协作手册。开始工作时先读这里，用它判断工作方式、文件边界、编码约束和最低验证要求。

写入规则：

- 写 `AGENTS.md`：只在新增或修改长期协作规则、全局硬约束、跨文件边界、常见踩坑经验、最低验证标准时写。不要写具体机制方案、临时判断、排查过程、TODO 或单次修改说明。
- 写 `docs/幕府系统设计文档.md`：当讨论或确定机制目标、功能定位、历史路线、系统关系、UI 承载方案、AI 设计方向时写。这里回答“这个机制为什么存在、要达到什么效果、和其他系统怎么配合”。
- 写 `docs/幕府机制场景测试用例.md`：当新增机制、修 bug、整理 scripted、修改事件/on_action/外交行动/和平条款/CB/政府改革/属国类型时写或更新。这里回答“Codex 在模拟器或工具里输入什么、查询什么、预期返回什么、重点检查什么”。

使用规则：

- 收到机制设计、路线规划、系统取舍类问题时，先看 `docs/幕府系统设计文档.md`，再讨论或修改设计。
- 收到实现、修 bug、整理 scripted、改事件或改外交行动类任务时，先看相关设计，再看 `docs/幕府机制场景测试用例.md` 是否已有场景；没有场景时先补测试用例，再改代码。
- 收到协作方式、编码规则、文件边界、验证标准、反复踩坑问题时，改 `AGENTS.md`，并检查测试用例文档是否需要承接对应验证方式。
- 如果三个文档内容重叠，优先拆分：规则留在 `AGENTS.md`，机制方案移到设计文档，可执行测试和模拟器操作移到测试用例文档。

## 测试用例与模拟器

测试用例和模拟器是本项目的长期验证主线，不是附属说明。

定位：

- `docs/幕府机制场景测试用例.md` 是 Codex 的可执行测试操作手册，不是给人进游戏照着点的手动说明。
- 测试用例的主体永远是 Codex：使用“我输入”“我查询”“我运行”“我检查”描述动作。
- 每个测试用例都应写清楚输入、查询、预期返回和重点检查。不要只写“模拟器做什么”或“玩家观察什么”。
- 模拟器是本 mod 逻辑的本地执行模型。随着机制推进和 bug 修复，它必须越来越贴近 EU4 与本 mod 的真实脚本执行链。

使用顺序：

- 新增机制或修 bug 时，先写或更新测试用例，再改模拟器或静态工具，最后改脚本。
- 测试用例必须先描述坏状态和期望状态，例如“我输入：A 使用 `po_mf_subjugate_daimyo` 收服 B”“我查询：B 目前的宗主”“预期：A”。
- 修复后必须运行对应模拟器函数或静态检查；如果没有对应工具，先补工具或明确记录为“我待补函数”，不能假装已经覆盖。
- 如果 bug 复发，第一步不是补兜底逻辑，而是补一个能复现坏状态的测试输入和查询断言。

模拟器规则：

- 模拟器模拟的是本 mod 当前脚本内容，不是凭感觉写出的理想机制。
- mod 里有的 subject type、government reform、flag、modifier、CB、war goal、peace treaty、scripted effect，模拟器才能使用；mod 里没有的 key 不得臆造。
- 模拟器执行一个机制时，应尽量按本 mod 真实脚本顺序模拟：入口、作用域、事件目标、条件筛选、效果调用、属国类型、政府改革刷新和清理 effect。
- 对关键链路，模拟器应先读取或静态确认本 mod 文件中存在对应 key 和效果调用，再模拟状态变化。
- 彼岸、Maid、原版和用户手写模组可以用来校验 EU4 作用域、CB、事件或 effect 写法，但模拟器最终执行和断言的对象必须是本 mod 的实际 key 和脚本链。
- 静态检查负责防止脚本结构回退，例如 CB 作用域写反、事件未排除本 mod 属国、私战收服错误增加幕府统御。模拟器负责验证状态流转是否符合预期。

测试文档写法：

- 步骤写输入动作：`我输入：建立 SHO、A、B`，`我输入：A 使用某和平条款收服 B`。
- 查询写明确对象：`我输入：查询 B 目前的宗主`，`我输入：查询 XIA 是否提供 cb_mf_xiayi_campaign`。
- 预期写返回值：`返回 A`，`返回 mf_retainer_daimyo_vassal`，`返回是/否`。
- 重点检查写验证边界：真实 key、真实效果顺序、不能只验证主目标而漏掉下级属国、不能把目标作用域写成进攻方作用域。

## 项目边界

这是一个《欧陆风云 IV》幕府体系扩展模组，当前目录名为“幕府修改0.1”，目标版本为 `v1.37.5.0`。

具体目标、核心功能、未来路线和机制方案以 `docs/幕府系统设计文档.md` 为准，`AGENTS.md` 不重复记录。

全局边界：

- 不能把本模组写成单纯削弱日本、强行禁止统一或隐藏吞并大名的脚本集合。
- AI 默认维持长期幕府-大名秩序；玩家仍应保留主动统一、改革和制度转向空间。
- 任何跨文件机制都必须服务可见治理链路：入口清楚、效果清楚、验证场景清楚。

## AI 行为边界

AI 幕府的默认目标是维持长期幕府-大名秩序，而不是最优速通统一。

约束：

- AI 默认不走速通统一路线。
- 合一、吞并、强中央集权逻辑必须有明确门槛，不能靠普通 AI 权重自然点满。
- AI 自动化只做“幕府会自然做的治理动作”，不要通过隐藏效果直接吞并大名、绕过陪臣层级或绕过玩家可见规则。
- AI 机制尽量少用月脉冲；能用外交行动、事件、决议、on_action 或共享 effect 表达时，先比较这些入口。

修改 AI 行为后，要关注日本是否又回到 50 年内过早统一的问题。

## EU4 脚本准则

写脚本前优先查原版、Maid mod、用户手写模组和 wiki，不要凭感觉编 key。

遇到不清楚的机制、作用域、命令、AI 行为或 UI 链路时，先看原版内容和参考 mod，再动手写。需要同时核对多个文件、多个来源或大段移植链路时，优先用子代理并行查看原版、Maid mod、用户手写模组和本 mod 当前实现，节约时间并减少漏项。

常用资料：

- 修正列表：`https://eu4.paradoxwikis.com/Modifier_list`
- 条件/触发器：`https://eu4.paradoxwikis.com/Triggers`
- 命令/效果：`https://eu4.paradoxwikis.com/Commands`
- 作用域：`https://eu4.paradoxwikis.com/Scopes`
- 中文百科“指令”：`https://www.eu4cn.com/wiki/指令`
- 中文百科“作用域”：`https://www.eu4cn.com/wiki/作用域`

常见原则：

- `trigger`、`potential`、`allow`、`limit` 中通常写条件。
- `effect`、`immediate`、事件选项中通常写效果。
- 区分 trigger scope 与 effect scope 中的 `any_*`、`all_*`、`every_*`、`random_*`。
- `limit = { }` 多用于 effect scope 中筛选对象。
- 事件若需要玩家或 AI 选择，不要做成 hidden 多选项事件；EU4 隐藏事件会默认走第一个选项。
- 修改 `topbar.gui`、`custom_gui`、`scripted_effects`、`new_diplomatic_actions`、`peace_treaties`、`wargoal_types`、`cb_types` 后，必须检查花括号数量。
- 修改后优先检查 `Documents/Paradox Interactive/Europa Universalis IV/logs/error.log`。
- 注释风格参考 Maid mod：复杂 effect、变量计算、作用域跳转、外交行动判断、UI flag 切换处写短中文注释；简单自明的单行不要硬注释。

## 本地化

Codex 在本地化中的全部工作只限于写入中文字符和补齐 key 文本。

重要约定：

- 需要新增或修改本地化时，可以在用户指定的中文存档文件和根目录 `localisation/*.yml` 外部本地化文件中直接写中文字符。
- Codex 不允许执行、建议执行或代替执行任何本地化编码转换工作。
- Codex 可以按用户要求在外部本地化文件中补中文 key 文本；这不等于执行转码。不要手动改写已有转码字符本身，除非用户明确要求替换该文本。
- 一切编码、转码、转换格式、进游戏读取格式处理都只能由人类完成。
- 外交行动、事件、修正、opinion modifier、tooltip、AI 接受理由、CB、战争目标、和平条款都要补完整本地化文本。

## 编辑防污染规则

这条规则优先级很高，用来避免继续制造编码混乱和难维护脚本。

- 项目脚本注释只写中文，不写英文注释。
- 打开或读取含中文注释的文件时，必须显式指定编码读取，例如 PowerShell 使用 `Get-Content -Encoding UTF8`。
- 修改含中文注释的文件前，必须先检查是否有 BOM、是否已有乱码、是否存在“注释和代码粘在同一行”的痕迹。
- 保存含中文注释的脚本文件时，禁止使用会隐式改编码或重写整文件的方式。不得用 `Set-Content`、`Out-File`、重定向、脚本批量读写等方式保存；手工小改只用补丁，只替换目标行。
- 修复乱码时只处理可确认语义的注释。不能确认原意的乱码注释，应删除或改成简短中文说明，不猜长段文本。
- 对转码敏感文件，只做最小脚本改动：不重排大段代码，不批量格式化，不移动无关块，不把整段逻辑重写成另一种风格。
- 如果必须修改转码敏感文件，优先复用已有 effect/trigger；不要新写 scripted trigger。

## 后续编辑准则

- 讨论阶段不要擅自改文件；用户明确说“修”“改”“处理”“更新”时可以直接动手。
- 用户只是问“能不能”“怎么做”时，默认只给方案，不自动落盘实现。
- 改动范围要小，不做无关格式化。
- 保留有价值的中文注释，但不要把临时排查、粗糙点、个人草稿写进项目说明。
- 如果文档与代码不一致，以代码和用户最新说明为准，再把文档修成原则性总结。
- 写功能机制时，先写或更新 `docs/幕府机制场景测试用例.md` 中对应的完整测试用例，明确步骤、预期和重点检查，再按测试用例实现机制。
- 修 bug 时，第一判断应是测试用例没有覆盖到真实坏状态。先把能复现 bug 的场景、坏状态、验证断言补进测试用例，再按测试用例修改代码；如果 bug 仍未解决，继续补测试用例，直到完全理解问题来源和修复边界。
- 新机制禁止继续塞进 `BL_*` 移植大文件和已乱码脚本里；优先放到 `mf_*` 命名的新文件或新 effect 中，旧文件只保留必要入口调用。
- 小模组优先追求简单、直接、可读的脚本链路。不要为了理论兜底写复杂 OR、标记链、死国身份推断或多层防御判断；除非已经有进游戏验证过的实际需求，否则入口条件和效果都用最少必要条件表达。
- Codex 不允许新增 scripted trigger。简单条件、单次使用条件、按钮/任务/外交行动专用条件直接写在调用处；不要为了“看起来整齐”抽成脚本化触发器。
- 清理已有 scripted trigger 前，必须先列出每个触发器的定义、调用位置、调用次数、用途判断和“是否由用户已有机制使用”，等待用户确认后才能删除或展开。不得擅自删除用户已经写好并正在使用的 scripted trigger。
- 能直接执行的效果不要包无意义判断，例如可直接 `change_government = monarchy`、直接清理相关政府改革，再添加目标改革；不要为了“可能已经是该状态”额外嵌套多层 `if`。
- 对已灭亡国家核心不要依赖当前政府改革、属国类型或历史身份兜底来判断；简单机制优先只筛 `exists = no`，再由恢复事件统一纳入幕府体系。
- 写测试和修 bug 时，先检查场景主链路是否清楚：入口、筛选、执行、事件、属国类型、政府改革和刷新 effect。不要用更宽的兜底掩盖链路断点。

## 验证清单

每次整理后至少检查：

- 文件是否被加 BOM。
- 是否引入新的乱码或注释粘连。
- 花括号数量是否平衡。
- `error.log` 是否出现新增错误。
- 按钮、事件、修正、CB、战争目标、和平条款是否有裸 key。
- AI 是否能执行预期路径。
- 玩家是否能看懂效果和代价。
- 是否违背“长期幕府-大名秩序”的设计目标。

涉及事件、on_action、custom_gui、scripted_effects、new_diplomatic_actions、peace_treaties、wargoal_types、cb_types 时，必须说明是否需要进游戏查看 `error.log`。
